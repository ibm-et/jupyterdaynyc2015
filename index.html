First off, a clarification

Every new Jupyter Subproject does not have to go through the incubator!

When to incubate

Another clarification

You are not required to incubate in github.com/jupyter-incubator!

tl;dr

Project in scope + proposer on Council + Council agrees
→ create new github.com/jupyter repo

Project mature in open source + want to move to github.com/jupyter
→ proposal for incorporation

Project is immature + want association with Jupyter
→ proposal for incubation under github.com/jupyter-incubator

Project under incubation matures
→ proposal for incorporation

Dashboards

  • Goal: ease friction between notebook use and app dev
  • Features: arrange output areas, capture layout, deploy apps
  • Recent focus: tutorials, demos, defects since open sourcing
  • Long term: maturation of deployment (Thebe, kernel gateway, tmpnb / binder)

Declarative Widgets

  • Goal: exploit web components ecosystem to ease the creation of rich interactive areas
  • Features: declarative syntax for function / data access, simple templates and data bindings, installation of 3rd party widgets
  • Recent focus: set and watch data from kernel language, import progress, error handling and display
  • Long term: more languages, better dev experience, clear identification of core modules (comm, manager, model, etc.)

Kernel Gateway

  • Goal: Websocket interface to kernels
  • Features: /api/kernels, /api/kernelspecs
  • Recent focus: Python reference implementation, hack notebook to launch remote kernels
  • Long term: use by dashboards, Thebe, Notebook, new kernel consumers, cloud kernel providers?
